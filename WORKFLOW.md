## 🎯 핵심 원칙

### 1. 한 번에 하나씩
"이거 다 만들어줘" ❌ → "M1의 공개 페이지 레이아웃만 만들자" ✅

큰 덩어리를 한 번에 시키면 버그가 어디서 생긴 건지 찾기 어려워요.

### 2. 마일스톤 기반으로 진행
`CLAUDE.md` 하단의 마일스톤 M1 → M2 → M3 → M4 → M5 순서대로.

### 3. 매 단계마다 실제로 확인
Claude가 "완료했습니다" 해도, 브라우저에서 직접 열어보고 확인하세요.

### 4. 모르는 건 물어보기
Claude Code에게 "이 코드가 무슨 뜻인지 쉽게 설명해줘"라고 언제든 물어도 돼요.

## 📅 일일 작업 흐름

### 작업 시작할 때

```bash
cd ~/Desktop/hey-kamori
claude
```

첫 메시지 예시:

> "CLAUDE.md와 SPEC.md 읽고, 어제 작업한 내용 요약해줘. 오늘은 M2의 링크 CRUD 모달을 만들자."

Claude Code는 **세션이 시작될 때마다 `CLAUDE.md`를 자동으로 읽어요**. 하지만 명시적으로 "먼저 읽어달라"고 말하면 더 확실해요.

### 작업 중

- 코드가 바뀔 때마다 `pnpm dev`가 돌고 있는 브라우저 새로고침
- 에러가 나면 **전체 에러 메시지 복붙**해서 Claude에게
- 디자인이 이상하면 스크린샷 찍어서 Claude에게 (터미널에 이미지 붙여넣기 가능)

### 작업 종료할 때

```bash
/save
```

슬래시 명령어로 현재 진행 상황을 저장하거나, Claude에게 "오늘 한 작업 요약하고 내일 이어서 할 수 있게 TODO.md 업데이트해줘"라고 말하세요.

그 후 **반드시 커밋**:

```bash
git add .
git commit -m "feat: 링크 목록 드래그 정렬 추가"
git push
```

(Claude Code가 대신 해주기도 해요. "커밋하고 푸시해줘"라고만 말하면 됨.)

## 💬 Claude에게 말하는 방법 (좋은 예 vs 나쁜 예)

### ❌ 나쁜 예

> "링크트리 만들어줘"
> → 너무 광범위. Claude가 뭘 우선할지 모름.

> "이거 안 되는데?"
> → 뭐가 안 되는지 모름. 에러 메시지 없음.

> "예쁘게 해줘"
> → 추상적. 기준 없음.

### ✅ 좋은 예

> "SPEC.md의 `/admin` 페이지 설명대로 편집 영역 레이아웃을 만들자. 미리보기는 다음 단계에서 할 거니까 지금은 건너뛰어."

> "링크 저장 버튼 누르면 이 에러가 떠: [에러 전체 복붙]. `app/admin/page.tsx`를 확인해줘."

> "DESIGN_TOKENS.md 기준으로, 링크 버튼 호버 시 살짝 위로 올라가는 효과를 추가해줘."

### 💡 팁

- **파일 경로를 명시**하면 Claude가 헤매지 않아요 (`app/admin/page.tsx`)
- **SPEC/DESIGN_TOKENS를 언급**하면 일관성이 지켜져요
- **"왜 이렇게 했어?"**라고 물어서 이해가 안 되면 "다시 설명해줘"

## 🚨 위험 신호 (이럴 땐 멈춰!)

아래 상황이면 **작업을 멈추고 물어보세요**:

| 신호 | 대응 |
|---|---|
| Claude가 `node_modules/` 안 파일을 수정한다고 함 | "거기 만지지 마" |
| 갑자기 10개 이상 파일을 한 번에 바꾸자고 함 | "한 파일씩 하자" |
| `.env.local`을 Git에 올리자고 함 | "거긴 절대 안 돼" |
| 기존 DB 컬럼을 지우자고 함 | "지우지 말고 DB_SCHEMA.md 먼저 보자" |
| "어차피 작동 안 하니까 새로 만들자" | "왜 작동 안 하는지 먼저 찾자" |

## 🔐 절대 하면 안 되는 것

1. **`.env.local` 파일을 GitHub에 올리기** — API 키 유출
2. **Claude가 생성한 SQL을 보지도 않고 `pnpm db:push`** — DB 날아감
3. **`git push --force`** — 팀 작업 중이면 이력 삭제됨 (혼자면 OK)
4. **Supabase 대시보드에서 직접 테이블 수정** — 마이그레이션과 불일치
